# Migration Discipline

> Team conventions for Supabase database migrations in this project.

---

## 1. Migration Naming Convention

Migrations are auto-generated by Lovable Cloud with timestamps:

```
YYYYMMDDHHMMSS_<uuid>.sql
```

**Current format**: `20260308212321_b4b4ff9c-f1ea-422d-bc70-2f24983ef1bc.sql`

### Manual migration naming (if needed outside Lovable)

```
YYYYMMDDHHMMSS_<action>_<entity>.sql
```

Examples:
- `20260310120000_create_consent_log.sql`
- `20260310130000_add_archived_status_to_jobs.sql`
- `20260310140000_add_withdrawn_status_to_applications.sql`

**Rules**:
- Timestamp prefix ensures execution order
- Action: `create`, `add`, `alter`, `drop`, `fix`, `seed`
- Entity: table name or feature name
- No spaces or special characters

---

## 2. Order & Dependency Rules

### Execution order

Migrations run in **timestamp order** (lexicographic sort). Never backdate migrations.

### Dependency handling

| Dependency Type | Rule |
|-----------------|------|
| New table with FK to existing | OK — create in one migration |
| New column with FK | Create column first, add FK constraint in same migration |
| Cross-table FK | Target table must exist (earlier migration or same migration, target first) |
| RLS policy referencing function | Create function first, then policy |
| Trigger referencing function | Create function first, then trigger |

### Same-migration ordering (within one file)

```sql
-- 1. Types/enums first
CREATE TYPE ... ;

-- 2. Functions before triggers/policies that use them
CREATE OR REPLACE FUNCTION ... ;

-- 3. Tables
CREATE TABLE ... ;

-- 4. Indexes
CREATE INDEX ... ;

-- 5. RLS enable + policies
ALTER TABLE ... ENABLE ROW LEVEL SECURITY;
CREATE POLICY ... ;

-- 6. Triggers
CREATE TRIGGER ... ;

-- 7. Grants (if any)
GRANT ... ;
```

## 3. Rollback Notes

**Every migration SHOULD include rollback comments** at the top:

```sql
-- Migration: Add archived status to jobs
-- Rollback:
--   ALTER TABLE jobs DROP CONSTRAINT IF EXISTS jobs_status_check;
--   -- Note: Cannot easily remove 'archived' from enum without recreating

-- Forward migration:
ALTER TABLE jobs ...
```

### Rollback feasibility matrix

| Change Type | Rollback Difficulty | Notes |
|-------------|---------------------|-------|
| Add column (nullable) | Easy | `ALTER TABLE DROP COLUMN` |
| Add column (NOT NULL) | Medium | Needs default or data migration |
| Drop column | Hard | Data loss; requires backup |
| Add table | Easy | `DROP TABLE` |
| Drop table | Hard | Data loss |
| Add enum value | Medium | Postgres can't remove enum values easily |
| Add RLS policy | Easy | `DROP POLICY` |
| Add function | Easy | `DROP FUNCTION` |
| Add index | Easy | `DROP INDEX` |

**CRITICAL**: Test rollbacks on staging before applying to production.

## 4. Seed-After-Migration Expectations

### When seeding is required

| Migration Type | Requires Seed? | Notes |
|----------------|----------------|-------|
| New lookup table | Yes | Populate enum-like data |
| New feature table | Maybe | Only for demo data |
| Schema-only change | No | |
| New required column | Maybe | Backfill existing rows |

### Seeding approach

1. **Lovable Cloud approach**: Use edge function `seed-demo-data` or `reset-demo-data`
2. **Manual approach**: Separate seed file or INSERT in migration (for lookup data only)

### Seed data location

```
supabase/functions/seed-demo-data/    → demo/test data
supabase/functions/reset-demo-data/   → reset to known state
```

**Rule**: Migrations should NOT contain demo data. Only reference/lookup data (enum tables) belongs in migrations.

## 5. Source of Truth Hierarchy

### Authoritative order (most → least)

1. **`src/domain/models.ts`** — Canonical TypeScript types
2. **`supabase/migrations/`** — Executed SQL schema
3. **`src/integrations/supabase/types.ts`** — Auto-generated (read-only)
4. **`docs/supabase-schema-plan.md`** — Design documentation
5. **`docs/domain-model.md`** — Conceptual documentation

### Sync discipline

| When | Action |
|------|--------|
| Adding new entity | 1) Update `domain/models.ts` → 2) Create migration → 3) Types auto-regenerate |
| Changing field | 1) Update `domain/models.ts` → 2) Create migration → 3) Update mappers |
| Schema drift suspected | Compare `domain/models.ts` against DB schema via Cloud UI |

### Type mapping

| Domain (TS) | Database (SQL) | Mapper location |
|-------------|----------------|-----------------|
| `camelCase` | `snake_case` | `src/domain/mappers.ts` |
| `string` enum | `text` or `enum` | `src/domain/models.ts` defines allowed values |
| `Date` / ISO string | `timestamptz` | ISO strings in code |
| `number[]` | `integer[]` | Direct |
| `string[]` | `text[]` | Direct |
| `object` | `jsonb` | Typed interfaces in models |

## 6. Pre-Migration Review Checklist

Before applying any migration:

### Schema review

- [ ] Does the change align with `src/domain/models.ts`?
- [ ] Are all new columns nullable OR have sensible defaults?
- [ ] Are foreign keys correct (no orphan risk)?
- [ ] Is RLS enabled on new tables?
- [ ] Are RLS policies complete (SELECT, INSERT, UPDATE, DELETE as needed)?

### Security review

- [ ] No `SECURITY DEFINER` without `SET search_path = public`
- [ ] RLS policies use `auth.uid()` correctly
- [ ] No accidental public access to sensitive data
- [ ] Check for privilege escalation (see `docs/rls-plan.md`)

### Compatibility review

- [ ] Will existing code break? (check repository files using this table)
- [ ] Are mappers updated for new/changed columns?
- [ ] Does the UI handle the new state/fields?

### Rollback review

- [ ] Is rollback documented in the migration?
- [ ] Is rollback tested on staging?
- [ ] Is there data that needs backup before applying?

## 7. Migration Workflow

### Lovable Cloud workflow (recommended)

```
1. Describe change in chat → Lovable generates migration
2. Review SQL in migration tool popup
3. Click "Run migration" (applies to test DB)
4. Verify in Cloud UI → Database tab
5. Test application against new schema
6. Publish → migration applies to live DB
```

### Manual workflow (outside Lovable)

```
1. Create migration file: supabase/migrations/YYYYMMDDHHMMSS_<name>.sql
2. Test locally against test DB
3. Review with team
4. Apply via Supabase CLI or dashboard
5. Update domain/models.ts if needed
6. Commit all changes together
```

## 10. Don'ts

- ❌ Don't edit `src/integrations/supabase/types.ts` — it's auto-generated
- ❌ Don't backdate migration timestamps
- ❌ Don't put demo data in migrations
- ❌ Don't create tables without RLS
- ❌ Don't use `SECURITY DEFINER` without `SET search_path`
- ❌ Don't drop columns/tables without backup plan
- ❌ Don't apply migrations to production without staging test
